# 大文档局部编辑实现方案（HaoMD）

## 1. 背景与问题

当前 HaoMD 在中小文档（数百行内）表现较好，但在“极大文档”（几千行、包含大量代码块/公式/图片）下存在明显卡顿：

- 在编辑器中连续输入字符时，能感知到明显的输入延迟；
- 拖动选择多行时，视口重绘成本较高，感觉“拖动发黏”；
- 预览、Outline 等已经有防抖和优化，但**单纯 CodeMirror 在大文档上的重排/重绘成本**仍然很高。

现有架构是：

- `WorkspaceShell` 维护整篇文档 `markdown`，并将其作为受控值传入 `EditorPane`（`CodeMirror`）和 `PreviewPane`（`MarkdownViewer`）；
- `CodeMirror` 始终持有完整文档，任何一次输入/删除都在整篇文档的上下文中进行；
- 预览端 `MarkdownViewer` 现在不再虚拟化（出于体验考虑），整体性能可接受。

在此基础上，**如果还想进一步优化“极大文档”的编辑体验，又不想破坏当前功能**，就需要一种“只在局部使用重型编辑器，高频操作不要牵动整篇文档”的架构。

## 2. 目标与约束

**目标**

- 在极大文档（如：> N 行，暂定 800 行）下，显著改善：
  - 连续输入字符时的输入延迟；
  - 拖动多行选择时的视口重绘卡顿。
- 对中小文档保持现有体验 **不变**。

**约束**

- **不影响现有功能**：
  - 多标签、自动保存 (`useFilePersistence`)；
  - 预览 (`PreviewPane`/`MarkdownViewer`)、Outline (`useOutline`)；
  - AI Chat、全局记忆、图片粘贴等基于“整篇 markdown 字符串”的功能。
- 行为上尽量保持对用户透明：
  - 打开小文档时完全感知不到模式变化；
  - 打开大文档时，编辑体验有所不同（局部编辑），但整体心智仍然是“在编辑同一篇文档”。
- 代码层面采用**特性开关 + 渐进式启用**，方便回滚和 A/B 测试。

## 3. 总体思路：全局只读 + 局部编辑

对极大文档，引入两层视图：

- **全局只读视图（Global Read-Only View）**
  - 依旧由 `MarkdownViewer` 渲染整篇文档；
  - 提供滚动、查看、跳转、Outline 定位等功能；
  - 简单高亮当前“可编辑块”，但不直接承担复杂编辑逻辑。

- **局部可编辑窗格（Local Editable Window）**
  - `CodeMirror` 不再持有整篇 markdown，只持有“当前编辑块”的子串；
  - 这个块可以是：当前段落、当前标题下的一节，或用户在预览/Outline 中点选的一段范围；
  - 用户在该窗口进行全功能编辑：输入、删除、选中、AI 插入代码等；
  - 修改完成后，将局部子串 merge 回全局 markdown，并刷新只读视图。

整体行为可以理解为：

> **整篇文档始终作为“源真相”保存在 `WorkspaceShell` 的 `markdown` 中，而 `CodeMirror` 只负责当前编辑片段，对这一片段做高性能编辑。**

## 4. 关键概念与数据结构

### 4.1 模式切换：普通模式 vs 大文档局部编辑模式

在 `WorkspaceShell` 层引入模式枚举：

```ts
// 只在内部使用，无需导出
type EditorMode = 'normal' | 'huge-doc-local-edit'
```

以及每个 tab 级别的状态结构（可以挂在 `WorkspaceShell` 的本地状态或 `tabs` 的 meta 上）：

```ts
type HugeDocState = {
  enabled: boolean              // 此 tab 是否启用局部编辑模式
  threshold: number             // 触发条件（行数阈值），便于调试
  currentChunk: {
    from: number                // 全局文档中的字符起始 offset（含）
    to: number                  // 全局文档中的字符结束 offset（不含）
    value: string               // 当前块的文本内容
  } | null
}
```

触发规则（初版建议）

- 打开文件或切换 tab 时：统计文档行数或长度：
  - 行数 > `HUGE_DOC_LINE_THRESHOLD`（例如 2000 行） ⇒ `enabled = true`；
  - 否则 `enabled = false`，使用当前的 `normal` 模式。
- 后续可以增加用户设置项（在 Settings 中暴露阈值和开关）。

### 4.2 文本片段映射（Chunk Mapping）

为了不改变 `useFilePersistence`、AI 功能等对“整篇 markdown 字符串”的依赖，我们只在编辑路径上使用局部片段，**所有持久化、AI 读取仍然基于 `WorkspaceShell.markdown`**。

需要一组纯函数来管理片段与全局文本的关系（可以放在 `app/src/modules/editor/chunkEdit.ts` 之类的工具模块）：

```ts
// 根据当前 activeLine / 光标位置，从整篇 markdown 中切出一个块
function extractChunkAroundLine(
  doc: string,
  line: number,
  options: { contextLines: number; maxLines: number },
): { from: number; to: number; value: string }

// 将局部编辑后的块合并回整篇文档
function applyChunkPatch(
  doc: string,
  chunk: { from: number; to: number },
  newValue: string,
): string

// 将局部块中的行号映射回全局行号（用于预览高亮）
function localToGlobalLine(
  doc: string,
  chunkFromOffset: number,
  localLine: number,
): number
```

注意：

- 片段不一定按行对齐，可以按最近的段落/标题边界对齐，这部分可以复用 Outline 的解析逻辑；
- `contextLines` 控制上下文范围，例如当前标题前后各 50 行，`maxLines` 控制单块上限，例如 300 行以内。

## 5. 与现有组件的集成方式

### 5.1 WorkspaceShell 调度逻辑

在 `WorkspaceShell` 里，当前是：

- `markdown` 作为 `EditorPane` 的 `value`；
- `handleMarkdownChange` 直接更新整篇 `markdown` + `updateActiveContent`；
- `activeLine`/`previewActiveLine` 驱动预览高亮；
- Outline、AI Chat 等都基于 `markdown`。

在局部编辑模式下，只在 `EditorPane` 这一层做分支：

- 保持 `WorkspaceShell.markdown` 始终是**整篇文档**；
- 为 `EditorPane` 计算一个 `editorValue`：
  - 普通模式：`editorValue = markdown`（不变）；
  - 局部编辑模式：`editorValue = hugeDocState.currentChunk?.value ?? markdown`（首块或退化为整篇）；
- 重写 `handleMarkdownChange` 的行为：

```ts
const handleMarkdownChange = useCallback((val: string) => {
  if (!hugeDocState.enabled || !hugeDocState.currentChunk) {
    // 保持现有逻辑
    setMarkdown(val)
    markDirty()
    updateActiveContent(val)
    return
  }

  // 局部编辑：只更新当前 chunk，然后合并回整篇文档
  const { from, to } = hugeDocState.currentChunk
  const nextFullDoc = applyChunkPatch(markdown, { from, to }, val)

  setMarkdown(nextFullDoc)
  markDirty()
  updateActiveContent(nextFullDoc)

  // 更新当前 chunk 的边界（长度可能变化）
  setHugeDocState((prev) =>
    prev && prev.currentChunk
      ? {
          ...prev,
          currentChunk: {
            from,
            to: from + val.length,
            value: val,
          },
        }
      : prev,
  )
}, [...])
```

> 这样可以确保：外部所有依赖 `markdown` 的逻辑（自动保存、AI、Outline、预览）都拿到的是最新的“整篇文档”。

### 5.2 EditorPane / CodeEditor 的行为

在 `EditorPane` 中：

- 接收 `editorValue`（整篇或 chunk）；
- 继续用当前的 `CodeEditor`，无需修改内部实现；
- `onCursorChange` 仍然上报局部行号，`WorkspaceShell` 再通过 `localToGlobalLine` 映射回全局行号，更新 `activeLine`：

```ts
<EditorPaneLazy
  markdown={editorValue}
  onChange={handleMarkdownChange}
  onCursorChange={(localLine) => {
    if (!hugeDocState.enabled || !hugeDocState.currentChunk) {
      setActiveLine(localLine)
      return
    }
    const globalLine = localToGlobalLine(markdown, hugeDocState.currentChunk.from, localLine)
    setActiveLine(globalLine)
  }}
  ...
/>
```

### 5.3 MarkdownViewer / PreviewPane

预览端原则：

- **始终渲染整篇 markdown**（不切片）；
- 利用现有的 `activeLine` 高亮当前行；
- 在局部编辑模式下，额外高亮当前可编辑块的行范围（可以通过传入 `currentChunkRange` 给 `MarkdownViewer`，用 CSS 做背景框）。

这样：

- 用户在预览中滚动、跳转、看全文，不受局部编辑影响；
- 看得见“当前正在编辑的是哪一段”。

### 5.4 Outline / 点击跳转逻辑

现有的 Outline 行为：

- 点击大纲项：`handleOutlineSelect` 通过 `scrollEditorToLineCenter` 滚动编辑器到对应行；
- `scrollEditorToLineCenter` 是基于整篇文档的行号和 `EditorView.state.doc` 的行结构。

在局部编辑模式下：

- 点击 Outline 时：
  1. 利用 `extractChunkAroundLine(markdown, targetLine, ...)` 生成一个以该标题/段落为中心的新 chunk；
  2. 更新 `hugeDocState.currentChunk`；
  3. 更新 `editorValue` 和 `CodeMirror` 内容（视作一次“大跳转”）；
  4. 再根据新 chunk 中对应的 local 行号，调用 `scrollEditorToLineCenter`（或直接设置 selection）让该标题在局部编辑器中置中。

=> 效果上，对用户来说还是“点大纲 → 编辑区跳到该部分”，只是编辑区只展示那一块。

### 5.5 AI Chat / 全局记忆 / 图片粘贴等

由于所有这些功能都调用了 `WorkspaceShell` 中的：

- `getCurrentMarkdown`（整篇）；
- `getCurrentSelectionText`（基于 `EditorView` 当前选区）；

在局部编辑模式下：

- `getCurrentMarkdown` 不变，仍返回整篇 markdown；
- `getCurrentSelectionText` 返回的是 chunk 内选区文本；
  - 如果 AI 插入内容（例如 `registerEditorInsertBelow` / `registerEditorReplaceSelection`），当前逻辑就是对 CodeMirror 的文本操作，然后 `syncEditorToReactState` 把完整文本写回 `markdown`；
  - 在局部模式下，只需要让 `syncEditorToReactState` 使用 `handleMarkdownChange` 的“局部 patch”逻辑即可。

由于这些操作都通过 `handleMarkdownChange` / `syncEditorToReactState` 最终落在整篇文本上，整体语义不会变，只是局部编辑器的内容代表了其中一部分。

## 6. 不影响现有功能的策略

为保障安全落地，本方案引入以下保护措施：

1. **特性开关 + 阈值**
   - 全局常量或配置：`ENABLE_HUGE_DOC_LOCAL_EDIT`（默认关闭，开发/测试阶段开启）；
   - 行数阈值：`HUGE_DOC_LINE_THRESHOLD`，只对超过阈值的文档启用局部模式；
   - 小文档入口完全不变。

2. **退化路径完备**
   - 如果在 `extractChunkAroundLine` 或 `applyChunkPatch` 中出现任何异常，立刻退化为 `normal` 模式：
     - 将 `editorValue` 切换为整篇 `markdown`；
     - 重置 `hugeDocState.enabled = false`；
     - 不阻塞用户继续编辑。

3. **不侵入核心逻辑**
   - `useFilePersistence`、AI 模块、Outline 解析、图片粘贴、AI Chat 等全部基于 `markdown` + `EditorView` 的公共接口，只在 `WorkspaceShell` <→ `EditorPane` 之间插入局部编辑逻辑；
   - 不修改 Tauri 后端、文件服务等模块。

4. **逐步启用**
   - 第 1 阶段：纯粹实现数据结构和工具函数，在内部埋点输出日志，不改变 UI（仍然展示整篇）；
   - 第 2 阶段：在开发构建中切换 `ENABLE_HUGE_DOC_LOCAL_EDIT = true`，手动在极大文档下验证；
   - 第 3 阶段：确认无数据损坏/行为异常后，才考虑在正式发布版本打开开关。

## 7. 实施步骤建议

### 步骤 1：抽象文本片段工具函数

- 新建 `app/src/modules/editor/chunkEdit.ts`：
  - 实现 `extractChunkAroundLine` / `applyChunkPatch` / `localToGlobalLine` 等；
  - 编写针对中英文混排、空行、代码块等情况的单元测试。

### 步骤 2：在 WorkspaceShell 中增加 HugeDoc 状态

- 添加：
  - `const [hugeDocState, setHugeDocState] = useState<HugeDocState | null>(null)`；
  - 在 `openFileInNewTab` / `applyOpenedContent` / `useEffect([markdown])` 中，当检测到文档行数超过阈值时，初始化 `hugeDocState`（但先不改变 Editor 行为，只输出调试日志）。

### 步骤 3：接入局部 editorValue（仅开发模式）

- 在传给 `EditorPaneLazy` 时，用：
  - `editorValue = hugeDocState?.enabled && hugeDocState.currentChunk ? hugeDocState.currentChunk.value : markdown`；
- 修改 `handleMarkdownChange`，按“普通模式”与“局部模式”分支处理；
- 暂时在 UI 上给出简易标记（如状态栏显示“局部编辑模式”）。

### 步骤 4：行号与预览高亮映射

- 在 `onCursorChange` 分支中加入 `localToGlobalLine` 映射，保持预览高亮仍然指向整篇文档对应位置；
- 可选：在 `MarkdownViewer` 中增加对“当前块范围”的高亮（例如用 CSS 背景轻微标出）。

### 步骤 5：Outline / 预览点击到块切换

- 修改 `handleOutlineSelect` 和 `handlePreviewLineClick`：
  - 普通模式：保持现有 `scrollEditorToLineCenter` 行为不变；
  - 局部模式：先 `extractChunkAroundLine`，更新 `hugeDocState.currentChunk`，然后在局部编辑器内滚动到对应 local 行。

### 步骤 6：AI 相关操作验证

- 重点验证：
  - AI 插入文本、替换选中内容时，整篇 `markdown` 正确更新；
  - `getCurrentMarkdown` / `getCurrentSelectionText` 在局部模式下仍然语义正确；
  - 自动保存不会丢失局部修改。

### 步骤 7：性能与回退

- 在极大文档下，验证：
  - 连续输入、拖选多行的流畅度；
  - 切换到其它 tab 再切回来时，局部块是否正确恢复；
  - 在开发构建中保留一个快捷开关（例如环境变量或调试命令）随时关闭局部模式。

---

通过以上方案，HaoMD 在保持现有功能不变的前提下，可以在极大文档上引入“全局只读 + 局部可编辑”的架构，将 CodeMirror 的重负载限制在当前编辑片段上，从而显著改善输入和选区操作的手感。
