下面给出按模块和优先级排序的重构规划（兼顾可读性、可维护性、性能与稳定性）。优先级从高到低，工作量为粗略人天预估。

### App 及布局/交互层（高优先）
- **待优化模块**: `App.tsx`（主布局、状态管理、快捷键、持久化）
- **重构措施**
  - 抽离布局状态与拖拽逻辑为自定义 hook（如 `useWorkspaceLayout`），隔离 `effectiveLayout`/`gridTemplateColumns`/拖拽事件和 localStorage 同步。
  - 抽离保存/快照/最近文件为 `useFilePersistence` hook，统一自动保存、手动保存、冲突处理与 Tauri 依赖的判断，避免重复调用。
  - 精简快捷键处理：将菜单触发与本地按键去重逻辑封装，明确 Tauri/非 Tauri 分支，避免重复触发。
  - 将预览/编辑器面板拆分为小组件（PreviewPane、EditorPane、Divider）以降低 `App.tsx` 体积。
  - 为布局相关数据增加类型约束与常量（如 `LAYOUTS`, `WIDTH_MIN/MAX`），减少魔法数。
- **理由**: 降低主组件复杂度，减少副作用耦合，提升可读性和可测试性，避免双重事件触发。
- **优先级**: P0
- **工作量**: 2–3 人日
- **风险**: 拆分时可能引入状态同步遗漏；需回归拖拽、快捷键、持久化。

### Markdown 渲染与组件层（高优先）
- **待优化模块**: `MarkdownViewer.tsx`、自定义 renderer 管理（`modules/markdown/plugins.ts`）、样式（`App.css` 中 Markdown/diagram 相关部分）
- **重构措施**
  - 去重代码块渲染逻辑（当前 mermaid/xmind 分支重复），抽象 `renderCodeBlock(lang, content)`。
  - 提供 renderer 注册时的类型收敛与错误兜底，避免无 renderer 时重复 fallback。
  - 为 active block 高亮/滚动逻辑增加节流/守卫，避免频繁 reflow。
  - 将 Markdown 样式拆出独立样式文件，减少 App.css 体积与耦合。
- **理由**: 提高可维护性和可扩展性，减少重复逻辑与渲染抖动。
- **优先级**: P0
- **工作量**: 1–2 人日
- **风险**: 渲染行为变化需验证所有自定义 code block；高亮/滚动节流可能影响体验。

### 图形渲染（Mermaid / Mind-elixir）（中高优先）
- **待优化模块**: `components/diagrams.tsx`
- **重构措施**
  - 将 Mermaid 与 Mind-elixir 逻辑分离为两个独立组件文件，提取通用的 resize/fit/hooks（如 `useResizeFit`）。
  - 明确缓存策略 key（代码+尺寸），增加过期或内存上限，防止长时间使用内存增长。
  - 抽出可配置项：padding、min/max scale，放入配置文件并集中管理。
  - 为 ResizeObserver/fit 增加防抖，避免高频回调；补充空容器/异常兜底。
- **理由**: 降低单文件复杂度，控制内存与渲染次数，便于调参。
- **优先级**: P1
- **工作量**: 1–2 人日
- **风险**: 缩放/居中行为变化需回归；过度防抖可能导致尺寸更新延迟。

### 文件持久化与 Tauri 交互（中优先）
- **待优化模块**: `modules/files/service.ts`、`autoSave.ts`、Tauri 命令交互
- **重构措施**
  - 统一 Tauri 可用性检查与错误映射，封装 `invokeSafe`，减少重复 try/catch。
  - 自动保存配置与“未命名”状态控制集中化，允许动态启停 autosave，并暴露手动 flush。
  - 对 `filesConfig` 进行环境区分（dev/prod），并提供日志/trace id 注入点。
  - 在前端层面对路径合法性与大小做预检，减少后端往返。
- **理由**: 提升可靠性与可调试性，减少意外重建/重复保存。
- **优先级**: P1
- **工作量**: 1–1.5 人日
- **风险**: 需回归保存/快照/最近文件流程；配置变更可能影响现有行为。

### 样式与设计体系（中优先）
- **待优化模块**: `App.css`（体积大、关注点混杂）
- **重构措施**
  - 拆分 CSS：布局/面板、Markdown、图形、弹窗/侧栏，按模块 import。
  - 通过 CSS 变量或 design tokens 管理颜色/间距/圆角，统一暗色主题。
  - 针对 grid 布局与 pane，增加 min-width/overflow 策略，减少内容撑破。
- **理由**: 提升可读性与一致性，便于主题/样式调整。
- **优先级**: P2
- **工作量**: 1–2 人日
- **风险**: 样式回归范围大；需检查滚动、对话框、分栏在不同尺寸下表现。

### 结构与可测试性（中低优先）
- **待优化模块**: 组件与 hooks 目录结构、测试空缺
- **重构措施**
  - 目录重组：`components/` 下按功能分子目录（editor, preview, diagrams, layout），`hooks/` 存放共享 hook。
  - 为关键逻辑添加轻量单元/组件测试（布局计算、fit 算法、文件服务 mapper）。
  - 为快捷键/菜单分发写最小集成测试或 story，用于手动回归。
- **理由**: 长期可维护性与回归效率。
- **优先级**: P2
- **工作量**: 2–3 人日
- **风险**: 引入测试依赖与基线维护成本。

### 性能与体验细项（低优先）
- **待优化点**: Markdown 预览节流、拖拽事件绑定/解绑、滚动同步
- **重构措施**
  - 调整 markdown 预览 debounce 时间可配置，视文档大小动态调整。
  - 为拖拽监听添加 passive/节流，确保卸载时清理。
  - 滚动同步增加容错与上限，避免大文档频繁 reflow。
- **理由**: 优化大文档场景体验。
- **优先级**: P3
- **工作量**: 0.5–1 人日
- **风险**: 参数调整可能改变交互手感。

### 风险总览
- 行为变更回归风险：布局、保存、图形缩放需全流程验证。
- 内存/性能回退风险：缓存策略和防抖节流需在大文件和窗口频繁 resize 场景下验证。
- 样式回归风险：CSS 拆分易导致选择器优先级问题，需视觉回归。
