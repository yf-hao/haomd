### 方案 1：命令系统抽离（建议优先）

- **目标**: 将命令表、命令分发与菜单/快捷键监听从 `App.tsx` 抽离，形成可复用的命令系统。
- **涉及文件**: `app/src/App.tsx`、`app/src/modules/commands/registry.ts`（新）、`app/src/hooks/useCommandSystem.ts`（可选新建）。
- **实施步骤**
  1. **提取类型与工厂**：把 `App.tsx` 顶部的 `AppCommand`、`CommandRegistry`、`CommandContext`、`createCommandRegistry` 复制到新文件 `modules/commands/registry.ts`，并对外导出。
  2. **收敛依赖**：在新模块中只保留“通过 `CommandContext` 间接调用 UI/服务”的方式，不直接引用 React 或具体组件。
  3. **精简 App 内部代码**：在 `App.tsx` 中改为 `import { createCommandRegistry } from './modules/commands/registry'`，保留现有 `useMemo` 注入上下文的写法。
  4. **抽出 Hook（可选）**：新建 `hooks/useCommandSystem.ts`，内部封装 `useMemo(commands)` 与 `dispatchAction`，并在其中挂载 `menu://action` 与快捷键监听；`App` 只负责调用 `useCommandSystem(context)` 并拿到 `dispatchAction`。
  5. **回归验证**：验证菜单点击、快捷键（保存/打开/新建/切换预览）功能是否与原行为一致。

### 方案 2：平台事件封装（Tauri Menu & Clipboard）

- **目标**: 将 `listen('menu://action')`、`listen('menu://open_recent_file')`、`listen('native://paste')` 等 Tauri 事件集中封装，减少 UI 组件对平台细节的感知。
- **涉及文件**: `app/src/App.tsx`、`app/src/modules/platform/menuEvents.ts`（新）、`app/src/modules/platform/clipboardEvents.ts`（新）、`app/src/hooks/useNativePaste.ts`（可选新建）。
- **实施步骤**
  1. **封装菜单事件**：新建 `modules/platform/menuEvents.ts`，提供 `onMenuAction(cb)` 和 `onOpenRecentFile(cb)`，内部使用 `listen('menu://action')` / `listen('menu://open_recent_file')`，并处理 StrictMode 下重复监听的问题。
  2. **封装剪贴板事件**：新建 `modules/platform/clipboardEvents.ts`，提供 `onNativePaste(cb)` / `onNativePasteError(cb)`，内部用现有的 `disposed` 方案封装 Tauri 粘贴事件。
  3. **Hook 层包装**：新建 `hooks/useNativePaste.ts`，接收 `editorViewRef` 与 `setStatusMessage`，内部调用 `onNativePaste` 等，将编辑器插入逻辑集中到一个 Hook 中。
  4. **替换 App 逻辑**：在 `App.tsx` 中删除直接调用 `listen('menu://action')` 和 `listen('native://paste')` 的 `useEffect`，改为调用新的 Hook/服务接口（例如 `useCommandSystem` 内部用 `onMenuAction`）。
  5. **回归验证**：在 Tauri 环境下验证菜单、最近文件、粘贴行为；在非 Tauri 环境下确认不会抛出异常。

### 方案 3：App Shell UI 组件化

- **目标**: 降低 `App.tsx` 的 UI 复杂度，把侧边栏、对话框等拆成独立组件，提高可读性与复用性。
- **涉及文件**: `app/src/App.tsx`、`app/src/components/RecentPanel.tsx`（新）、`app/src/components/ConflictModal.tsx`（新）。
- **实施步骤**
  1. **抽取最近文件侧边栏**：新建 `components/RecentPanel.tsx`，将 `App` 中 `<aside className="side-panel recent-panel">...` 相关 JSX 与交互逻辑移动过去，通过 props 传入 `open`、`recent`、`recentHasMore`、`recentLoading`、`onLoadMore`、`onOpenItem`、`onDeleteItem`、`onClose` 等。
  2. **抽取冲突弹窗**：新建 `components/ConflictModal.tsx`，接收 `conflictError`、`onRetrySave`、`onCancel` 等 props，将当前模态框 JSX 与事件处理迁移过去。
  3. **App 中只负责拼装**：`App.tsx` 中仅保留 `<RecentPanel ... />`、`<ConflictModal ... />` 的使用，逻辑集中在组件内部，减少 `App` 的行数和嵌套层级。
  4. **样式适配**：确保新组件继续复用原有 CSS class（如 `recent-panel`、`modal` 等），避免大规模样式改动。
  5. **回归验证**：检查最近文件打开/删除、冲突弹窗的显示与按钮行为是否与原先一致。

### 方案 4：文件持久化与 Tauri 适配下沉

- **目标**: 让 `useFilePersistence` 聚焦“文件领域逻辑”，把 `isTauri`、`invoke('set_title')`、`openDialog`/`saveDialog` 等平台相关细节抽到独立服务中，提高可移植性和开闭性。
- **涉及文件**: `app/src/hooks/useFilePersistence.ts`、`app/src/modules/platform/tauriDialogService.ts`（新）、`app/src/modules/platform/tauriWindowService.ts`（新）、`app/src/modules/files/recentService.ts`（可选新建）。
- **实施步骤**
  1. **封装窗口标题服务**：创建 `tauriWindowService.ts`，导出 `setWindowTitle(title: string)`；内部处理 `isTauri` 判断与 `invoke('set_title')`，对错误统一打日志但不抛出到 UI。
  2. **封装对话框服务**：创建 `tauriDialogService.ts`，导出 `openMarkdownFileDialog()` 和 `saveMarkdownFileDialog(suggested: string)`，内部封装 `openDialog`/`saveDialog` 和 Tauri 可用性检查，将错误统一映射为 Result 类型。
  3. **改造 useFilePersistence**：用这些服务替换直接调用 `invoke` 和 `plugin-dialog` 的代码，使其只处理“何时保存/打开”与“如何更新状态”，不关心 Tauri 的细节。
  4. **（可选）抽出最近文件服务**：将 `clearRecentRemote`、`deleteRecentRemote`、`listRecentPage`、`logRecentFile` 等封装到 `recentService.ts`，`useRecentFilesState` 只依赖这个服务，从而与具体后端实现解耦。
  5. **回归验证**：测试打开/保存/另存为/自动保存/最近文件相关功能，确认行为与 Tauri 环境下的预期一致。

### 方案 5（可选）：状态码与文案解耦

- **目标**: 把文件子系统中的错误与状态由“直接存中文文案”改为“状态码 + 文案映射表”，为后续国际化和统一处理打基础。
- **涉及文件**: `app/src/hooks/useFilePersistence.ts`、`app/src/modules/files/types.ts`、`app/src/modules/files/errorMessages.ts`（新）。
- **实施步骤**
  1. **定义错误/状态枚举**：在 `types.ts` 中增加 `FileErrorCode`、`FileStatus` 等枚举或字面量类型，将常见错误（如 CONFLICT、INVALID_PATH、UNKNOWN 等）与状态（saving/saved/error/conflict）统一命名。
  2. **调整 Result 类型**：让服务层返回的 `Result` 中包含 `code` 而不是直接的中文 message。
  3. **新增文案映射表**：在 `errorMessages.ts` 中根据 `FileErrorCode` 映射到中文提示（或将来支持多语言），UI 层通过该表展示消息。
  4. **改造 useFilePersistence**：内部使用状态码 + 映射表设置 `statusMessage` 和 `saveStatus`，避免在 hook 中散落硬编码文案。
  5. **回归验证**：检查各种异常情况下的提示文案是否正确显示，保证用户体验不下降。
