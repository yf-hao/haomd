## HaoMD 启动性能优化实施步骤

> 本文档基于当前项目结构（React + Tauri + CodeMirror + Markdown 预览）编写，目标是改善应用 **冷启动到 Welcome 页** 以及 **首次进入工作区** 的速度。每个步骤都说明需要修改的文件、建议的改动方式和注意事项。

---

## 步骤 0：基线测量（只读观察，不改功能）

**目的**：先了解当前启动性能数据，为后续优化提供对比基线。

### 需要关注的文件/位置

- `app/src/App.tsx`（用于临时埋点）
- Tauri DevTools → Performance / Network 面板（运行时工具，不涉及代码）

### 操作建议

1. **在 `App.tsx` 顶部增加简单的时间戳（开发环境用）**
   - 在文件顶部靠近 `const isTauri = () => ...` 前后添加：

   ```ts
   const appStartTime = performance.now()
   ```

2. **在 `App` 组件的 `return` 前后增加一次日志**（确认首屏渲染大致完成时间）

   ```ts
   console.log('[Perf] App first render cost:', performance.now() - appStartTime, 'ms')
   ```

3. **使用 Tauri DevTools 的 Performance 面板录制一次启动过程**
   - 打开应用 → 立刻开始录制 → 等 Welcome 页稳定后停止。
   - 记录：
     - JS 解析与执行大致耗时。
     - 首次 commit / 首次 paint 时间。

> 这些埋点可在后续正式做优化时移除或包裹在 `import.meta.env.DEV` 条件内。

## 步骤 1：拆分 App，增加 `WorkspaceShell`（欢迎页与工作区解耦）

**目的**：冷启动阶段只挂轻量的 Welcome UI，把编辑器、预览、文件持久化、多标签逻辑等重组件延后到“进入工作区”时再初始化。

### 需要修改/新增的文件

- 新增：`app/src/components/WorkspaceShell.tsx`（建议位置）
- 修改：`app/src/App.tsx`

### 1.1 新增 `WorkspaceShell` 组件

**文件**：`app/src/components/WorkspaceShell.tsx`

**主要职责**：
- 管理当前的 **工作区状态**：
  - 布局：`useWorkspaceLayout`
  - 标签：`useTabs`
  - 文件持久化：`useFilePersistence`
  - 命令系统：`useCommandSystem`
  - Sidebar / Outline：`useSidebar`、`useOutline`
  - 粘贴 / Tauri 菜单 / 退出逻辑：`useNativePaste`、`onOpenRecentFile`、`handleQuit` 等
- 渲染：
  - `TabBar`
  - `EditorPane` + `PreviewPane`
  - `Sidebar` / `OutlinePanel`
  - 分割条（布局拖拽）

**实现思路**：

1. 从 `App.tsx` 中剪切以下逻辑到新组件：
   - 所有和 tab 有关的状态及函数：
     - `useTabs(...)` 调用以及导出的 `tabs`, `activeId`, `activeTab`, `createTab`, `setActiveTab`, `closeTab`, `closeCurrentTab`, `getUnsavedTabs`, `updateActiveContent`, `updateActiveMeta`。
     - `closeCurrentTabRef`、`handleCurrentTabClose`、`handleTabClose`、`handleTabSaveAndClose`。
   - 布局相关：
     - `useWorkspaceLayout` 返回的 `layout`, `setLayout`, `showPreview`, `setShowPreview`, `dragging`, `workspaceRef`, `effectiveLayout`, `gridTemplateColumns`, `previewWidthForRender`, `startDragging`。
     - `main.workspace` 的布局 JSX 片段（包含 `EditorPane` / `PreviewPane` / 分割条）。
   - 文件持久化相关：
     - `useFilePersistence` 调用及解构出的全部字段。
     - `persistenceOptions`、`handleManualSave`、`applyOpenedContent`、`openFileInNewTab`、`openFileFromSidebar`、`openRecentFileInNewTab`、`openFolderInSidebar`、`closeTabsByPath`。
     - 以及 `ConflictModal`、`confirmDialog`、`quitConfirmDialog` 的渲染逻辑。
   - Sidebar / Outline：
     - `useSidebar` 调用及 `sidebar` 变量。
     - `useOutline(markdown)` 和 `outlineItems`。
     - `activeOutlineId`, `handleOutlineSelect`。
     - `Sidebar`、`OutlinePanel`、`handleSidebarContextAction`。
   - 命令系统 / 退出逻辑：
     - `useCommandSystem` 调用及 `dispatchAction`。
     - `handleQuit` 函数。
   - 粘贴 / 编辑器滚动：
     - `useNativePaste(editorViewRef, setStatusMessage)`。
     - `scrollEditorToLineCenter`、`editorViewRef`。

2. 在 `WorkspaceShell` 中定义一个 Props 接口，用于与外层 `App` 交互，例如：

   ```ts
   export interface WorkspaceShellProps {
     onExitToWelcome?: () => void
   }
   ```

   - 如果将来需要从工作区回到 Welcome，可以用 `onExitToWelcome` 通知外层。

3. `WorkspaceShell` 内部保持原有的 `return` JSX 结构（`TabBar` + `main.workspace` + 模态框），尽量少改动行为。

> 建议先“原样搬运”逻辑，保证功能正常，再做后续的懒加载和优化。

### 1.2 精简 `App.tsx`，负责模式切换

**文件**：`app/src/App.tsx`

1. 保留的状态与逻辑：
   - `LeftPanelId` / `activeLeftPanel` 及其切换按钮（活动栏）。
   - 顶层的 `app-shell` 布局和底部状态栏。
   - 判断是否 Tauri 环境的 `isTauri`（可留在此文件或抽到 util）。

2. 新增一个状态表示是否进入工作区，例如：

   ```ts
   const [hasWorkspace, setHasWorkspace] = useState(false)
   ```

3. 在 `Welcome` 组件的使用处（原有的 `tabs.length === 0 ? <Welcome ...` 部分）改为：
   - 在 `App.tsx` 中：
     - 当 `!hasWorkspace` 时渲染 `Welcome`，并把 `onNewFile` / `onOpenFile` 改为 **仅切换状态**：

     ```tsx
     <Welcome
       onNewFile={() => setHasWorkspace(true)}
       onOpenFile={() => setHasWorkspace(true)}
     />
     ```

   - 当 `hasWorkspace` 为 `true` 时，渲染 `WorkspaceShell`：

     ```tsx
     {hasWorkspace && (
       <WorkspaceShell
         // 将来如果需要，可以传入回调，比如 onExitToWelcome
       />
     )}
     ```

4. 原本依赖 `tabs.length` 判断是否显示 `Welcome` 的逻辑，迁移到 `WorkspaceShell` 内部：
   - 在 `WorkspaceShell` 内依然可以保留 `tabs.length === 0` 时展示 `Welcome` 或其他提示，但对外层 `App` 来说，只负责“是否已经进入工作区”。

> 完成本步骤后：冷启动时不会立即初始化编辑器、文件持久化、命令系统，能明显减轻首次 render 的负担。

## 步骤 2：对编辑器与预览做懒加载（代码分割）

**目的**：把 CodeMirror、Markdown 渲染、KaTeX、highlight.js 等重依赖拆到单独 chunk，只在进入工作区并真正需要时加载。

### 需要修改的文件

- `app/src/components/WorkspaceShell.tsx`（新增的工作区容器）
- 视情况微调：`app/src/components/EditorPane.tsx`、`app/src/components/PreviewPane.tsx`

### 2.1 在 `WorkspaceShell` 中对重组件使用 `React.lazy`

1. 在 `WorkspaceShell.tsx` 顶部引入 `lazy` 和 `Suspense`：

   ```ts
   import { lazy, Suspense } from 'react'
   ```

2. 用 `lazy` 包装重组件：

   ```ts
   const EditorPaneLazy = lazy(() => import('./EditorPane'))
   const PreviewPaneLazy = lazy(() => import('./PreviewPane'))
   ```

3. 在 JSX 中使用 `Suspense` 包裹原本的编辑器和预览区域：

   ```tsx
   <Suspense fallback={<div className="pane loading">加载编辑器...</div>}>
     <EditorPaneLazy
       markdown={markdown}
       onChange={handleMarkdownChange}
       onCursorChange={setActiveLine}
       showPreview={showPreview}
       setShowPreview={setShowPreview}
       editorViewRef={editorViewRef}
     />
   </Suspense>
   ```

   对 `PreviewPane` 同理：

   ```tsx
   <Suspense fallback={<div className="pane loading">加载预览...</div>}>
     <PreviewPaneLazy
       value={previewValue}
       activeLine={activeLine}
       previewWidth={previewWidthForRender}
       fullWidth={effectiveLayout === 'preview-only'}
     />
   </Suspense>
   ```

4. 根据布局不同（`preview-left`, `preview-right`, `preview-only`, `editor-only`），用懒加载版本替换原组件。

### 2.2 保持 `EditorPane` / `PreviewPane` 自身 API 不变

- 确保 `EditorPane.tsx` 和 `PreviewPane.tsx` 的导出是 **默认导出或具名导出** 与 `lazy` 一致：
  - 如果你使用 `export function EditorPane`，则 `import('./EditorPane')` 的 `default` 需要适配（可改为 `export default EditorPane` 或在 lazy 中使用 `.then` 转换）。
- 同样检查 `PreviewPane` 是否与 `lazy` 的期望一致。

> 完成后：CodeMirror、MarkdownViewer、KaTeX、highlight.js 所在的 bundle 会被拆分，只在真正进入工作区时异步加载。

## 步骤 3：调试日志与全局监听仅在开发环境启用

**目的**：减少生产环境多余的 JS 执行和事件监听，避免对启动与运行时性能造成不必要的影响。

### 需要修改的文件

- `app/src/hooks/useTabs.ts`
- `app/src/App.tsx`

### 3.1 `useTabs` 中的调试日志

**文件**：`app/src/hooks/useTabs.ts`

1. 包裹日志输出，使其只在开发环境生效，例如使用 Vite 的 `import.meta.env.DEV`：

   ```ts
   if (import.meta.env.DEV) {
     console.log('[useTabs] hook 已初始化')
   }
   ```

2. 对其它 `console.log`、`console.warn` 也使用同样模式包裹，或者统一抽成一个 `debugLog` 函数：

   ```ts
   function debugLog(...args: any[]) {
     if (import.meta.env.DEV) {
       console.log(...args)
     }
   }
   ```

   然后将原本的 `console.log` 替换为 `debugLog`。

### 3.2 `App.tsx` 中的全局 click 监听与日志

**文件**：`app/src/App.tsx` 或迁移后的 `WorkspaceShell.tsx`

1. 找到类似如下的 `useEffect`：

   ```ts
   useEffect(() => {
     const handler = (e: MouseEvent) => {
       // ... 全局点击日志
     }
     window.addEventListener('click', handler, true)
     return () => {
       window.removeEventListener('click', handler, true)
     }
   }, [])
   ```

2. 将整个 `useEffect` 包裹在开发环境条件中：

   ```ts
   if (import.meta.env.DEV) {
     useEffect(() => {
       // ... 保持原有逻辑
     }, [])
   }
   ```

   或者在内部第一行加判断：

   ```ts
   useEffect(() => {
     if (!import.meta.env.DEV) return
     // ... 原有逻辑
   }, [])
   ```

3. 对 `activeLine` 变化打印日志、标签关闭/退出流程中的大量日志，同样做条件限制或适当删除。

> 完成后，生产环境将不会注册这些与调试相关的监听与日志，有助于减小运行时开销。

## 步骤 4：依赖与资源瘦身

**目的**：减少首屏 bundle 体积，从构建产物上进一步压缩启动成本。

### 需要检查/修改的文件

- `package.json`
- 代码中对第三方库的引用（如 `react-syntax-highlighter`、`highlight.js`、KaTeX 等）

### 4.1 清理未使用的依赖

1. 打开 `package.json`，确认目前的依赖列表。
2. 在代码中搜索这些依赖的使用情况（例如查找 `react-syntax-highlighter` 的 import）：
   - 如果已经完全不使用某依赖，则：
     - 删除对应 import 语句。
     - 移除 `package.json` 中该依赖，后续执行 `npm install` / `pnpm install` 重新锁定依赖。

### 4.2 静态资源优化（可选）

1. 检查 `app/src/assets/logo.png` 的实际尺寸和体积。
   - 如果远大于 UI 需要的展示尺寸，可压缩/裁剪后替换。
2. 如果有其它大图、背景图，统一进行压缩处理。

> 这一部分对 Tauri 桌面应用更多影响包体体积和加载时间，对启动速度有一定帮助但不如结构优化直接。

## 步骤 5：大纲与 Markdown 解析的进一步优化（进阶，可择机实施）

**目的**：为超大 Markdown 文档提供更好的性能，避免在主线程一次性做太重的解析工作。

### 需要关注的模块

- `app/src/hooks/useOutline.ts`
- `app/src/modules/outline/parser.ts`
- `app/src/components/MarkdownViewer.tsx`

### 5.1 大纲解析延迟/节流

1. 在 `useOutline` 内部：
   - 对 `markdown` 变更使用 `debounce`（如 200–300ms），避免每次按键都重新解析大纲。
   - 或使用 `requestIdleCallback`（带回退），在浏览器空闲时进行解析。

2. 在 `WorkspaceShell` 中：
   - 确认只有在工作区存在至少一个 tab 时才启用大纲解析。

### 5.2 Web Worker 方案（仅在大文档下考虑）

1. 将 `modules/outline/parser.ts` 的解析逻辑抽象出纯函数接口（输入字符串，输出大纲数据）。
2. 新建 Worker 脚本，例如 `app/src/workers/outlineWorker.ts`，在其中调用该解析函数。
3. 在 `useOutline` 中通过 `new Worker(...)` 与 Worker 通信，主线程只负责接收解析结果并 setState。

> 这一步实现成本较高，优先级可以低于前 1–4 步，更多是为将来面对极大文档时预留优化空间。

## 步骤 6：构建与打包级优化（代码压缩与分包）

**目的**：在不引入复杂重构的前提下，通过 Vite 构建配置进一步减小首屏需要解析的 JS/CSS 体积，并让“重依赖”落在更合理的异步 chunk 中。

### 6.1 启用更激进的压缩选项（移除残留调试代码）

**文件**：`app/vite.config.ts`

当前配置非常精简：

```ts
export default defineConfig({
  plugins: [react()],
})
```

虽然 Vite 默认会对生产构建做压缩，但可以显式调整：

1. **确保生产构建使用压缩与现代目标**（一般已经默认）：

   ```ts
   export default defineConfig({
     plugins: [react()],
     build: {
       target: 'es2018',
       minify: 'esbuild', // 默认就是 esbuild，这里只是显式说明
     },
   })
   ```

2. **按需移除残留的 `console` / `debugger`**：
   - 你已经用 `import.meta.env.DEV` 包裹了主要调试日志，理论上生产环境 Tree Shaking 可以去掉大部分。
   - 如果将来仍有零散日志不方便统一处理，可以考虑在 `build` 里增加 esbuild 的 `drop` 选项：

   ```ts
   build: {
     minify: 'esbuild',
     target: 'es2018',
     // 可选：确保生产包中没有零散 console/debugger
     // esbuild: {
     //   drop: ['console', 'debugger'],
     // },
   },
   ```

   > 建议先依赖 `import.meta.env.DEV` 方案，只有当你确认不再需要生产日志时再开启 `drop`，避免影响真实错误日志排查。

### 6.2 使用 `manualChunks` 把重依赖拆成更明确的模块

从构建输出可以看到：

- `EditorPane-*.js` ~ 600KB（gzip ~200KB）；
- `PreviewPane-*.js` ~ 900KB（gzip ~300KB）；
- 还有一个较大的 `katex-*.js` 和 `cytoscape.esm` 相关 chunk。

这些体积主要来自：

- `@uiw/react-codemirror` + 一系列 `@codemirror/*`；
- Markdown 渲染栈：`react-markdown` + `remark-*` + `rehype-*` + `react-syntax-highlighter`；
- 图表相关：`mermaid`、`mind-elixir`、`cytoscape`；
- KaTeX：`katex` + 字体。

你已经通过懒加载把 `EditorPane` 和 `PreviewPane` 拆成独立 chunk，但可以通过 `manualChunks` 让这些依赖进一步按职责拆分，避免单个 chunk 过大：

**建议方案（示例配置）：**

```ts
export default defineConfig({
  plugins: [react()],
  build: {
    target: 'es2018',
    minify: 'esbuild',
    rollupOptions: {
      output: {
        manualChunks: {
          // 编辑器相关（CodeMirror + 封装）
          editor: [
            '@uiw/react-codemirror',
            '@codemirror/autocomplete',
            '@codemirror/commands',
            '@codemirror/lang-javascript',
            '@codemirror/lang-markdown',
            '@codemirror/language',
            '@codemirror/lint',
            '@codemirror/state',
            '@codemirror/theme-one-dark',
            '@codemirror/view',
          ],
          // Markdown 渲染和代码高亮
          markdown: [
            'react-markdown',
            'remark-gfm',
            'remark-math',
            'rehype-katex',
            'rehype-highlight',
            'react-syntax-highlighter',
          ],
          // 图表/思维导图等
          diagrams: [
            'mermaid',
            'mind-elixir',
            'cytoscape',
          ],
          // 数学公式渲染
          katex: ['katex'],
        },
      },
    },
  },
})
```

**效果预期：**

- 主入口和 `WorkspaceShell` 所在的 chunk 体积会更小，重依赖被划到 `editor` / `markdown` / `diagrams` / `katex` 等专用 chunk；
- 由于你已经对编辑器和预览做了懒加载，这些 chunk 只会在对应组件首次被加载时被请求（Tauri 场景是从本地磁盘读取）。

> 注意：`manualChunks` 是“偏好”配置，实际 chunk 划分还会受 Rollup/Vite 合并策略影响，建议配合 `rollup-plugin-visualizer` 或 Vite 的 `--profile` 工具观察效果。

### 6.3 引入 bundle 分析工具（辅助调优）

**目标**：更直观地看到哪些模块占用了最多体积，验证 `manualChunks` 的调整是否达到预期。

1. 安装可视化插件（任选其一）：
   - `rollup-plugin-visualizer`

2. 在 `vite.config.ts` 中加入插件：

   ```ts
   import { visualizer } from 'rollup-plugin-visualizer'

   export default defineConfig({
     plugins: [
       react(),
       visualizer({
         filename: 'dist/stats.html',
         open: false, // 构建后手动打开
         gzipSize: true,
         brotliSize: true,
       }),
     ],
   })
   ```

3. 运行 `npm run build` 后，打开 `dist/stats.html`：
   - 找出仍然占用较大体积的模块（例如某些不常用的图表类型）。
   - 决定是否进一步做“按需加载”或“条件编译”。

## 步骤 7：预览侧重依赖的更细粒度懒加载

**目的**：在现有“Editor/Preview 懒加载”的基础上，再把预览内部的重依赖拆得更细，比如 Mermaid / 思维导图 / 部分 KaTeX，避免在普通文档中也加载所有图表和高级特性。

### 7.1 图表（Mermaid / 思维导图）按需加载

当前结构（简化）：

- `MarkdownViewer.tsx` 顶部直接 `import { MermaidBlock, XMindBlock } from './diagrams'`；
- `diagrams.tsx` 顶部直接 `import mermaid from 'mermaid'`、`import MindElixir` 等。

这意味着：

- 只要加载 `PreviewPane` 所在的 chunk，就会一并加载 `mermaid` 和 `mind-elixir`，即使用户的文档里根本没有 ` ```mermaid` 或 ` ```mind` 代码块。

**优化方案（只改预览内部，不改外层架构）：**

1. 在 `MarkdownViewer.tsx` 内，把图表渲染器改为“懒组件”：
   - 使用 `React.lazy` + `Suspense`，在检测到某个代码块的语言是 `mermaid` 或 `mind` 时才异步加载 `./diagrams`：

   ```tsx
   const DiagramsLazy = React.lazy(() => import('./diagrams'))

   // 在 code 渲染器中：
   if (lang === 'mermaid' || lang === 'mind') {
     return (
       <Suspense fallback={<pre>Loading diagram...</pre>}>
         <DiagramsLazy.Renderer lang={lang} code={content} />
       </Suspense>
     )
   }
   ```

   - `./diagrams` 中导出一个统一的 `Renderer`，内部根据 `lang` 决定用 `MermaidBlock` 还是 `XMindBlock`。

2. 好处：
   - 普通 Markdown 文档（没有图表）不会加载 `mermaid`/`mind-elixir` 相关代码；
   - 图表使用者首次遇到图表时会有一个很小的加载延迟，但只在那一刻发生。

### 7.2 KaTeX / 代码高亮按需加载（可选）

1. **KaTeX**：
   - 目前 `katex/dist/katex.min.css` 是在 `App.tsx` 顶部全局引入的。
   - 可以考虑把 CSS 移动到 `MarkdownViewer.tsx` 顶部或 `PreviewPane` 模块中，让它随预览 chunk 一起懒加载，而不是在冷启动时就解析。

2. **代码高亮**：
   - 如果未来觉得 `react-syntax-highlighter` 体积仍然偏大，可以考虑：
     - 换成按语言拆分的轻量高亮库；
     - 或者按语言动态加载高亮样式（复杂度较高，优先级可以后置）。

> 这些属于“尾声优化”，核心是避免在典型文档场景下加载大量用不到的渲染器。

## 建议的“第二阶段”实施顺序

在已完成步骤 1–4 的基础上，如果你希望继续压榨启动性能，可以按以下顺序推进：

1. **先做 步骤 6.1 + 6.2（构建压缩 + manualChunks）**
   - 修改 `vite.config.ts`，显式设置 `build.target/minify`，并添加基础的 `manualChunks` 规则；
   - 用构建日志和可视化报告确认主入口 chunk 体积是否明显下降。

2. **再做 步骤 7.1（Mermaid / 思维导图按需加载）**
   - 改造 `MarkdownViewer` + `diagrams`，确保普通文档不加载图表相关代码；
   - 用 bundle 分析工具确认 `mermaid`、`mind-elixir` 是否被拆到单独 chunk。

3. **视需要执行 步骤 7.2（KaTeX/CSS 等进一步拆分）**
   - 将 `katex` CSS 移到预览模块，避免冷启动时解析；
   - 对高亮库的替换或懒加载在确认收益明显时再做。

4. **最后按需探索 步骤 8（Tauri 微调）和 步骤 5（Worker 化）**
   - 主要针对极大文档 or 低性能设备场景，作为后续长期演进的方向。

这样，你的文档就分成了两层：
- 第一阶段（步骤 0–5）：已经完成的结构性优化 + 基础瘦身；
- 第二阶段（步骤 6–8）：构建层面的压缩与按需加载精细化，专注于“重依赖分离”和“非常规场景”下的体验。
