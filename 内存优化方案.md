## HaoMD 内存优化方案（按优先级）

> 目标：在不牺牲核心功能体验的前提下，降低应用在大文档、长对话和长时间运行场景下的内存占用，并避免内存线性增长或极端峰值。

---

### P0：Markdown 预览内存优化（最高优先级）

#### 1. 虚拟滚动 / 分段渲染大 Markdown

- **现状**
  - `MarkdownViewer` 使用 `ReactMarkdown + remark-gfm + remark-math + rehype-katex`，整篇文档一次性渲染到 DOM。
  - 大文档时，AST + 虚拟 DOM + 高亮后的 DOM 同时驻留内存，滚动和编辑时频繁重排和重绘。

- **目标**
  - 对超大 Markdown 文档（按行数或字节数判断）采用“分段渲染 + 虚拟滚动”策略，只保留当前视口附近的内容在 DOM 中。

- **实施步骤**
  1. **增加文档体积检测**
     - 在进入 `MarkdownViewer` 时，根据 `value.length` 或按行数粗算文档规模。
     - 定义阈值，例如：`> 5000 行` 或 `> 500 KB` 即视为大文档，启用“虚拟滚动模式”。
  2. **基于块的渲染模型**
     - 预解析 `value`：按标题、空行或固定行数划分为多个“渲染块”（例如每块 200 行）。
     - 维护一个包含所有块的数组：`[{ id, startLine, endLine, text }]`。
  3. **视口感知 + 只渲染可见块**
     - 使用容器高度和滚动位置计算当前可见的块索引范围（可以用简单的自实现或引入 `react-window` 类库）。
     - 只对可见块调用 `ReactMarkdown` 渲染，其它块渲染为占位 div（仅撑高度，不含子树）。
  4. **与现有“行高亮”逻辑兼容**
     - 当前通过 `data-line-start` / `data-line-end` 实现高亮和滚动定位。
     - 在虚拟模式下，滚动至目标行所在块时，确保该块一定处于渲染范围内，再执行高亮逻辑。

- **验收标准**
  - 加载 1–2 MB 以上的 Markdown 时：
    - 首次渲染内存占用明显低于当前实现（JS heap 峰值下降）。
    - 滚动时卡顿减少，帧率更稳定。

---

#### 2. 代码块高亮懒加载（视口内才高亮）

- **现状**
  - 所有代码块默认使用 `react-syntax-highlighter`（Prism + oneDark）一次性高亮，DOM 数量多且样式复杂。
  - 大量代码块时，高亮部分占用大量 JS 和 DOM 内存。

- **目标**
  - 初始只渲染代码块为简单 `<pre><code>`；仅在用户滚动到可视区域附近时，才替换为高亮组件。

- **实施步骤**
  1. **拆分 code 渲染逻辑**
     - 在 `MarkdownViewer` 自定义 `code` 组件中：
       - 仍然解析出 `lang`、`content`。
       - 但默认返回一个“占位组件”，标记 `data-code-id` 和原始文本。
  2. **视口检测**
     - 在 `MarkdownViewer` 中维护一个 `visibleCodeIds` 集合。
     - 使用 `IntersectionObserver` 或滚动监听，检测哪些 `data-code-id` 的元素进入视口（加一些缓冲，比如上下各多 1–2 屏）。
  3. **按需挂载 SyntaxHighlighter**
     - 对 `visibleCodeIds` 内的代码块渲染为 `SyntaxHighlighter`。
     - 离开视口很远的代码块可以退回简单 `<pre><code>`，释放一部分 DOM（可选，视复杂度决定是否做）。

- **验收标准**
  - 含几十个以上长代码块的文档中，初次加载和滚动时内存与 CPU 压力明显减小。
  - 视口内的代码仍然有完整高亮效果，无明显闪烁。

---

#### 3. 图片懒加载与尺寸控制

- **现状**
  - `img` 渲染器目前统一设置 `style={{ maxWidth: '50%', height: 'auto', width: '300px' }}`，但未做懒加载。
  - 大量或大分辨率图片会在进入文档时一次性加载，占用内存。

- **目标**
  - 延迟加载不在视口的图片，并对极大图片做显示层面的限制，降低峰值内存。

- **实施步骤**
  1. 在自定义 `img` 组件中添加 `loading="lazy"` 属性。
  2. 对于超大图片（可通过自然宽高判断），采用“缩略图 + 点击放大”模式：
     - 初始仅显示限制尺寸的缩略版。
     - 用户点击时弹出单独的图像查看器组件，再加载原始大图。

- **验收标准**
  - 含大量图片的文档中，初次加载时网络和内存占用更平滑，滚动时图片逐步加载。

---

#### 4. 非活动标签页的预览卸载

- **现状**
  - 多标签页同时打开时，每个 tab 可能都保留一个完整的 `MarkdownViewer` 实例和渲染树。

- **目标**
  - 对于非当前活动的 tab，只保留原始 Markdown 文本，不保留渲染树；切回时再重新渲染。

- **实施步骤**
  1. 在上层 tab 管理组件中，区分 `active` 与 `inactive` tab。
  2. 对 `inactive` tab：
     - 只渲染一个轻量占位（如文件名、图标），不挂载 `MarkdownViewer`。
  3. 切换到某 tab 时再创建 / 挂载 `MarkdownViewer`，并恢复滚动位置（如果已有记录）。

- **验收标准**
  - 打开多篇大文档时，总内存占用随 tab 数量的增长速度显著降低。

### P1：AI Chat 会话内存优化

#### 1. 对话历史长度限制

- **现状**
  - `ConversationState` 中 `engineHistory` 和 `viewMessages` 会随着对话轮次线性增长。
  - 长对话时，同一内容在“引擎历史”和“展示消息”中存在重复字符串。

- **目标**
  - 限制保留在内存中的历史轮数，如只保留最近 20 条完整消息；旧消息可被截断或总结。

- **实施步骤**
  1. 在 `appendUserInput` 和 `appendAssistantPlaceholder/Chunk`（领域层）中加入“截断逻辑”：
     - 每次添加新消息后检查 `engineHistory` 长度，如果超过阈值，丢弃最旧的若干条（或合并为一条 summary）。
  2. 在构建 `viewMessages` 时保持与 `engineHistory` 同步，只展示最近 N 条。
  3. 如果未来需要“完整对话历史”，可以考虑使用后台持久化存储，而不是全部留在内存，但这属于后续扩展。

- **验收标准**
  - 长时间连续对话后（几十轮以上），内存占用趋于稳定而不是持续上涨。

---

#### 2. 历史结构瘦身与避免重复存储

- **现状**
  - `engineHistory` 和 `viewMessages` 分别维护一份文本内容。
  - 每次请求前通过 `engineHistoryToChatMessages` 构建新的数组，存在额外复制。

- **目标**
  - 降低历史结构的冗余，必要时通过轻量引用而不是复制字符串。

- **实施步骤**
  1. 评估是否可以以 `engineHistory` 为单一数据源：
     - UI 通过投影逻辑从 `engineHistory` 导出 `viewMessages`（不再存两份 content）。
  2. `engineHistoryToChatMessages` 改为：
     - 只选择最近 N 条消息构建 `messages`，避免构建全量历史；
     - 直接引用已经存在的字符串，避免多次拼接或复制。

- **验收标准**
  - 对话轮数增加时，memory profiler 中字符串数量和大小的增长速度减缓。

---

#### 3. 会话超时与资源释放

- **现状**
  - `ChatSession` 提供 `dispose()`，但释放时机目前依赖 UI 调用；长时间打开 AI Chat 窗口可能一直保留客户端和状态。

- **目标**
  - 在用户长时间不再使用 AI Chat 时自动释放底层客户端资源，避免闲置占用。

- **实施步骤**
  1. 在前端 `AiChatDialog` 中维护“最后交互时间戳”。
  2. 使用定时器或 `useEffect` 循环检查：
     - 如果距离上次交互超过阈值（比如 10 分钟），自动调用 `session.dispose()`，并在 UI 上标记为“已自动断开，需要时重新连接”。

- **验收标准**
  - 长时间打开应用但不使用 AI Chat 时，内存不会因为会话遗留对象而持续占用。

### P2：Tauri 后端（Rust）内存优化

#### 1. 大文件读写的内存边界控制

- **现状**
  - 已有 `MAX_FILE_BYTES = 20MB` 限制，防止单文件过大。
  - 读文件时 `Vec<u8>` 与 `String` 在函数生命周期内同时存在。

- **目标**
  - 避免在读写过程中产生不必要的额外缓冲，保证内存占用与文件大小近似线性。

- **实施步骤**
  1. 在读取文件时：
     - 继续使用当前逻辑即可，当前 `bytes` 是局部变量，函数结束即可释放；
     - 如有需要，可考虑边读边 hash，减少在极端情况下的峰值，但目前收益有限。
  2. 根据实际使用情况评估是否需要将 `MAX_FILE_BYTES` 调低（例如 10MB），进一步限制前端内存压力。

- **验收标准**
  - 打开接近上限大小的文件时，观察应用总内存不会出现异常峰值或剧烈波动。

---

#### 2. 目录遍历（`list_folder`）的结果规模控制

- **现状**
  - `collect_entries` 会递归遍历整个目录树，将所有符合条件的文件和目录都 push 到 `Vec<FsEntry>` 中。
  - 对极大仓库或目录树，`Vec<FsEntry>` 会比较大。

- **目标**
  - 对结果规模进行硬限制，防止在特殊目录下占用过多内存。

- **实施步骤**
  1. 为 `collect_entries` 增加参数：最大深度、最大条目数。
  2. 当条目数量超过阈值（例如 5000）时：
     - 停止继续递归。
     - 返回错误或特殊结果，提示用户目录过大需缩小范围。

- **验收标准**
  - 打开非常大的目录树时，后端内存占用保持在可控范围，且不会导致前端 UI 卡死。

### P3：整体策略与实施节奏

#### 1. 测量与基线

- 在实施任何代码改动前，建议先记录当前基线：
  - 使用浏览器 DevTools 的 Memory/Performance：
    - 加载一个大 Markdown（带多图、多 code、多公式），记录 JS heap 峰值和渲染时间。
    - 开启 AI Chat，进行 20–30 轮对话，记录内存变化曲线。
  - 在需要时使用系统级工具观察 Tauri 进程内存。

#### 2. 实施顺序建议

- 第一阶段：**P0**（Markdown 预览优化）
  - 先做“非活动标签页卸载”和“图片懒加载”（实现简单，回归风险低）。
  - 再推进“虚拟滚动 + 懒高亮”（需要更多测试，但收益最大）。
- 第二阶段：**P1**（AI Chat 内存优化）
  - 优先实现“对话历史长度限制”和“构建消息时只取最近 N 条”。
  - 若仍有需要，再考虑统一历史结构、自动超时释放 session。
- 第三阶段：**P2**（后端细节）
  - 根据实际使用场景，微调 `MAX_FILE_BYTES` 和目录遍历限制。

#### 3. 回归与验证

- 每个阶段完成后，重复第一节中的基线测试：
  - 确认内存占用合理下降。
  - 确认交互体验（滚动、编辑、AI 对话）未明显变差。

> 以上方案只涉及设计与实现步骤，未在代码中做任何修改，后续可以按优先级逐步落地。
