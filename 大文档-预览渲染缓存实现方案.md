## 大文档预览渲染缓存实现方案

### 1. 背景与目标

HaoMD 的预览链路基于 `ReactMarkdown + remark + 自定义组件`，在包含大量公式、代码块、图表和多媒体内容的大文档下，预览渲染和更新会明显占用 CPU，导致以下问题：

- 输入时预览掉帧、卡顿；
- KaTeX、Prism 高亮、Mermaid 图表等重组件重复执行相同的重计算；
- 文本发生局部修改时，未修改的块也会重复渲染。

本方案的目标：

- **对 KaTeX / 代码高亮 / 图表 / 图片 / 音视频等重组件做内容级缓存**；
- 在文档更新时，**仅对源内容发生变化的块执行重计算**；
- 尽量不改变现有的 `MarkdownViewer` 对外接口和使用方式。


### 2. 现有架构简要回顾

- 入口：`WorkspaceShell` 计算 `previewValue` / `previewActiveLine` → `PreviewPane`。
- 预览核心：`MarkdownViewer` 负责：
  - 使用 `ReactMarkdown` 解析 `value`；
  - remark 插件打行号锚点、处理 math；
  - 自定义组件渲染：
    - KaTeX：`math` / `inlineMath`；
    - 代码高亮：`components.code` + `Prism SyntaxHighlighter`；
    - 图表：`DiagramsLazy (mermaid / mind)`；
    - 图片 / 音频 / 视频：处理路径和协议 `haomd://`；
    - 折叠、行号映射、高亮 active block。

**关键点**：当前所有重组件都是“纯函数式”的：每次 `value` 变化，`ReactMarkdown` 重建 AST，所有组件按新的 props 运行一次，不做内容级缓存。


### 3. 缓存策略总览

#### 3.1 缓存粒度与缓存键

- **粒度**：以“组件级块”为单位，而不是整篇文档：
  - KaTeX 块公式：每个 `math` 节点；
  - KaTeX 行内公式：每个 `inlineMath` 片段；
  - 代码块：每个 `<pre><code>` / `SyntaxHighlighter` 调用；
  - 图表：每个 mermaid/mind 代码块；
  - 图片 / 音频 / 视频：每个媒体标签。
- **缓存键（Key）设计原则**：
  - 唯一标识一个组件的“渲染源内容”；
  - 相同源内容 → 命中缓存，不重复重算；
  - 源内容改变 → miss，执行重算并更新缓存。

常见键方案：

- KaTeX：`key = mode + '|' + tex`，其中 `mode ∈ {block, inline}`；
- 代码块：`key = lang + '|' + content`；
- 图表：`key = lang + '|' + code`；
- 媒体：
  - 图片：`key = finalSrc + '|' + maxWidth`；
  - 音频：`key = finalSrc`；
  - 视频：`key = finalSrc + '|' + posterUrl`。


#### 3.2 缓存位置与生命周期

- 缓存采用 **模块级 WeakMap / Map** 存储：
  - 适合在单文档会话中复用；
  - 页面刷新或关闭后自然释放；
  - 不持久化到磁盘。
- 对于 KaTeX 等可能持有大量字符串的缓存，可选：
  - 限制最大条目数（LRU 淘汰）;
  - 针对大文档预览单独配置 cache limit。


#### 3.3 与 React 渲染的配合方式

- 重组件统一封装成单独的 React 组件（或提取到 `components` 子模块），例如：
  - `CachedMathBlock` / `CachedInlineMath`；
  - `CachedCodeBlock`；
  - `CachedDiagramBlock`；
  - `CachedMediaImage` / `CachedMediaAudio` / `CachedMediaVideo`。
- 每个组件内部：
  - 用 `useMemo`/模块级 Map 做 **源内容 → 渲染结果** 的缓存；
  - 外层再用 `React.memo` 做 **props 比较**，避免无意义的 re-render。
- `MarkdownViewer` 的 `components` 映射只负责把 remark 节点拆成 props，交给这些缓存组件渲染。


### 4. KaTeX 缓存方案

#### 4.1 Block 公式（math）

- 现有逻辑：
  - 从 `value` 或 `node.value` 拿到 TeX 字符串；
  - `katex.renderToString(tex, { displayMode: true })`；
  - 用 `dangerouslySetInnerHTML` 插入 HTML；
  - 打上 `data-line-start` / `data-line-end` 参与折叠和高亮。

- 缓存改造：

  - 定义模块级缓存：
    - `const blockMathCache = new Map<string, string>()`;
  - 封装组件 `CachedMathBlock`：
    - 输入：`tex: string`, `start?: number`, `end?: number` 等；
    - 内部：
      - `key = 'block|' + tex`；
      - 查 `blockMathCache`，命中则直接用缓存的 HTML；
      - miss 时调用 `katex.renderToString`，将结果写入缓存再渲染；
    - 输出：包裹了 `data-line-*` 和 `LineRangeContext` 的 `<div>`。

- remark component 中：
  - 仅负责解析 `start/end` 和 `tex`；
  - 将这些 props 传给 `CachedMathBlock`，自身逻辑尽量简化。

#### 4.2 Inline 公式（inlineMath）

- 现有逻辑：
  - 尝试从 `node.value` 或 `props.value` 获取 TeX；
  - 使用 `katex.renderToString(tex, { displayMode: false })`；
  - 直接用 `<span dangerouslySetInnerHTML={{ __html: html }}>`。

- 缓存改造：

  - 模块级缓存：`const inlineMathCache = new Map<string, string>()`；
  - `CachedInlineMath`：
    - 输入：`tex: string`（以及可选的 lineRange，用于折叠判断）；
    - 键：`key = 'inline|' + tex`；
    - 同样逻辑：命中 → 用缓存；miss → 调 KaTeX 更新缓存。
  - 与 block 一样可以搭配 `React.memo` 做 props 比较，避免重复执行组件函数。


### 5. 代码高亮缓存方案

#### 5.1 现状

- 使用 `react-syntax-highlighter` + `Prism`：

  ```ts
  <SyntaxHighlighter
    language={lang}
    style={oneDark}
    showLineNumbers
    wrapLines
    {...rest}
  >
    {content}
  </SyntaxHighlighter>
  ```

- 每次 `value` 变化都会重新执行高亮解析，对多段代码块的大文档来说开销较大。

#### 5.2 缓存设计

- 模块级缓存：`const codeHighlightCache = new Map<string, React.ReactNode>()`；
- `CachedCodeBlock` 组件：
  - 输入：`lang?: string`, `content: string`, 行号信息（用于折叠）等；
  - 键：`key = (lang ?? 'plain') + '|' + content`；
  - 逻辑：
    - 命中：直接返回缓存的 ReactNode（注意 ReactNode 可重复挂载）；
    - miss：执行 `SyntaxHighlighter` 渲染，将节点树存入缓存。
  - 结合折叠：若块完全在折叠区域内，**在调用 `CachedCodeBlock` 之前就 return null**，避免生成 key 和缓存。

#### 5.3 区分轻重路径

- 对于 **单行、无语言** 的 code：仍然走最简单的 `<code>{content}</code>`，不进缓存；
- 仅对 **多行或指定了语言的 code block** 使用 `CachedCodeBlock` + 缓存。


### 6. 图表（Mermaid / Mind / XMind）缓存方案

#### 6.1 现状

- `components.code` 中：当 `lang` 为 `mermaid` / `mind` 时，使用 `DiagramsLazy` 动态渲染图表：

  - `React.Suspense` + `DiagramsLazy`；
  - 实际渲染由 `diagrams` 模块内的逻辑负责。

#### 6.2 缓存设计

- 缓存粒度：以 **同一 `lang` + 同一 `code`** 为单位。
- 缓存位置：
  - 在 `diagrams` 模块内部维护 `Map<string, DiagramRenderResult>`；
  - 结果可以是：SVG/Canvas 内容、布局数据等。
- 渲染策略：
  - `CachedDiagramBlock` 接收 `lang` 与 `code`；
  - `key = lang + '|' + code`；
  - 若缓存有对应的渲染结果，直接用缓存数据构建 React 结构；
  - miss 时调用 Mermaid/Mind 的布局算法，生成结果后写入缓存。

> 备注：Mermaid 本身有解析/布局缓存，具体可在 `diagrams` 内再利用其 API；方案层面只要求“不重复对相同 code 多次跑完整布局”。


### 7. 媒体组件（Image / Video / Audio）缓存方案

#### 7.1 Image

- 现状：
  - 解析 `alt` 中的宽度标记 `(30%) / (300px)`；
  - 根据 `filePath + src` 计算最终 `finalSrc`（含 `haomd://` 或 `https://haomd.localhost`）；
  - 渲染 `<img>`，开启 `loading="lazy"`。

- 缓存优化：
  - 图片本身的加载已经由浏览器缓存负责，CPU 开销不大；
  - 但路径解析（尤其是相对路径 → 绝对路径 → encode）在大文档中会重复执行。

- 方案：
  - 定义 `const imageUrlCache = new Map<string, string>()`；
  - `CachedImage`：
    - 键：`key = filePath + '|' + rawSrc`；
    - 若命中：直接用缓存的 `finalSrc`；
    - miss：执行路径转换、编码，写入缓存。
  - 渲染 `<img>` 部分仍由 React 正常 diff，利用浏览器的网络缓存减轻资源加载负担。

#### 7.2 Audio / Video

- 基本思路与图片相同：
  - 计算媒体资源路径时使用缓存；
  - 防止在同一文档中多次重复计算同一个相对路径 → 绝对路径 + 协议编码。
- 视频还有 `poster` 解析逻辑，可以同样缓存 `posterUrl`。


### 8. 组件整合与接入方式

#### 8.1 在 MarkdownViewer 中封装缓存组件

- 在 `MarkdownViewer.tsx` 顶部附近新增一组缓存组件声明（或拆到 `MarkdownViewer.cache.tsx`）：
  - `CachedMathBlock` / `CachedInlineMath`；
  - `CachedCodeBlock`；
  - `CachedDiagramBlock`（如果决定在这里包一层）；
  - `CachedImage` / `CachedAudio` / `CachedVideo`。
- 每个组件：
  - 负责确定缓存 key；
  - 内部使用 Map + `useMemo` 或模块级缓存获取/写入重计算结果；
  - 外层用 `React.memo` 封装，比较关键 props（如 `tex`/`content`/`src`）。

#### 8.2 替换 MarkdownViewer 的 components 映射

- 在 `components = useMemo(() => ({ ... }))` 中：

  - `math` → `CachedMathBlock`；
  - `inlineMath` → `CachedInlineMath`；
  - `code` 中：
    - 当 `lang` 为 `mermaid` / `mind` 时，调用 `CachedDiagramBlock`；
    - 其他多行 code → `CachedCodeBlock`；
  - `img` → `CachedImage`；
  - 音频/视频逻辑集中在媒体组件中，统一使用路径缓存。

> 注意：折叠逻辑（`isBlockFolded`）尽量放在调用缓存组件之前，如块被折叠则直接返回 `null`，避免产生无意义缓存。


### 9. 大文档下的配套策略

- 在 `WorkspaceShell` 中已有大文档判定逻辑（针对编辑）：
  - 预览层可以单独定义 `isHugeDocPreview` 标志，用行数判定；
  - 在大文档预览模式下：
    - 更依赖缓存组件的效果；
    - 可进一步提升 `markdown → previewValue` 的节流时间，减少整体 AST 重建频率。

- 缓存与大文档模式的配合：
  - 文档越大、公式/代码块越多，缓存命中率越高，收益越明显；
  - 行为上不会对用户暴露“模式切换”，只是预览更平滑。


### 10. 边界与风险评估

- **内存占用**：
  - 大量不同内容的代码块/公式/图表会增加缓存大小；
  - 可考虑为各类缓存设置最大条数（例如 500/1000），超出时按 LRU 淘汰。

- **一致性**：
  - 缓存键必须覆盖所有影响渲染的因素（例如 KaTeX 配置、主题、语言、高亮样式等），否则修改配置后可能命中旧缓存；
  - 初期可以只按内容缓存，如未来引入“可动态切换主题/高亮样式”，需要把配置也纳入 key。

- **复杂度**：
  - 方案在 `MarkdownViewer` 内部增加了一层缓存逻辑，但对外 API 不变；
  - 测试要覆盖：
    - 文档多次修改、撤销；
    - 同一内容出现多处；
    - 折叠/展开对缓存的影响（折叠不应该清空缓存）。


### 11. 实施顺序建议

1. **第一阶段：重组件缓存落地**
   - KaTeX block/inline；
   - 代码高亮（Prism）多行块；
   - Mermaid/Mind 图表；
   - 图片/音视频路径解析缓存。

2. **第二阶段：大文档预览节流调优**
   - 引入 `isHugeDocPreview` 标志；
   - 大文档下加大预览更新节流时间；
   - 观察 CPU 占用和交互流畅度。

3. **第三阶段：根据实际效果迭代**
   - 视情况增加 LRU 淘汰策略；
   - 如仍存在卡顿，再考虑更重的优化（预览虚拟化、worker/后端渲染）。
