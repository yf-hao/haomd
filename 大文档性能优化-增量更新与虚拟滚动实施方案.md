# 大文档性能优化：增量更新与虚拟滚动实施方案

## 1. 目标与背景
当前的预览方案在处理超过 5000 行的文档时，即使有内容缓存，React 的调和（Reconciliation）过程和海量的 DOM 节点依然会导致明显的输入延迟和滚动卡顿。

本方案旨在通过 **“片段化增量渲染 (Fragmented Incremental Rendering)”** 与 **“真正的虚拟滚动 (True Virtualization)”**，将渲染开销从 $O(N)$ 降低到 $O(1)$。

---

## 2. 核心设计思路

### 2.1 片段化 (Fragmenting)
不将整篇 Markdown 字符串交给单一的 `ReactMarkdown` 组件，而是将其拆分为多个 **Fragment（片段）**：
- 按空行或顶级块（H1-H6, Paragraph, CodeBlock）进行切分。
- 维护一个 `fragments: string[]` 数组。

### 2.2 增量更新 (Incremental Update)
- 当用户编辑文档时，定位到受影响的 `fragmentIndex`。
- 仅更新 `fragments` 数组中对应的索引项。
- 配合 `React.memo`，未修改的片段对应的 React 组件将直接跳过解析和渲染，响应时间从数百毫秒降低至 1ms 级。

### 2.3 真正的虚拟滚动 (True Virtualization)
- 使用 `virtua` 或 `react-virtuoso` 等支持动态高度的虚拟列表。
- 只有当前视口（Viewport）内的片段才会被真正挂载到 DOM 中。
- DOM 节点总数维持在 50-100 个，无论文档有几万行。

---

## 3. 详细实施步骤

### 第一阶段：预览层架构重构

#### 1. 引入虚拟化库
安装 `virtua`（轻量且支持动态高度同步）。
```bash
npm install virtua
```

#### 2. 实现 MarkdownFragment 组件
将现有的 `MarkdownViewer` 逻辑抽离出原子组件，仅负责渲染一小段 Markdown。
- **输入**: `content: string`, `lineOffset: number`, `sharedComponents: any`。
- **优化**: 使用 `React.memo`，依赖项仅为 `content`。

#### 3. 实现 VirtualMarkdownViewer 容器
- 负责将原始 `string` 还原为 `string[]`。
- 使用 `VList` 渲染片段列表。
- 维护 `lineMap` 用于双向滚动同步（行号 -> Fragment 索引）。

### 第二阶段：编辑器增量对接

#### 1. 优化数据流 (WorkspaceShell)
- 改进 `handleMarkdownChange`，计算出受影响的片段索引。
- 避免直接频繁设置全量大字符串，可考虑使用 `useReducer` 或不可变数据结构管理片段数组。

#### 2. 启用 HugeDoc 编辑器模式
- 重新审视 `setHugeDocEnabled(true)`。
- 编辑器侧只维持当前视口前后的内容，与预览侧的虚拟化形成闭环。

### 第三阶段：同步与交互适配

#### 1. 同步滚动 (Sync Scroll)
- 在虚拟化环境下，传统的 `getBoundingClientRect()` 无法获取不可见块的位置。
- 方案：通过虚拟列表提供的 `scrollToIndex` 或 `scrollToOffset` 进行同步。

#### 2. Active Line 高亮
- 仅当对应的 Fragment 在视口内时才应用 `active-block` 样式。
- 由于是以 Fragment 为单位，高亮精度将精确到块级。

## 4. 关键技术方案 (Code Snippets)

### Fragment 切分逻辑示例
```typescript
function splitMarkdownIntoFragments(markdown: string) {
  // 按空行切分，但保留连续的代码块或列表
  // 这里可以根据实际需求调整切分粒度
  return markdown.split(/\n\n+/); 
}
```

### 虚拟列表结构示例
```tsx
const MarkdownFragment = React.memo(({ content, components }) => (
  <ReactMarkdown components={components}>
    {content}
  </ReactMarkdown>
), (prev, next) => prev.content === next.content);

function VirtualViewer({ value }) {
  const fragments = useMemo(() => splitMarkdownIntoFragments(value), [value]);

  return (
    <VList>
      {fragments.map((frag, idx) => (
        <MarkdownFragment key={idx} content={frag} />
      ))}
    </VList>
  );
}
```

## 5. 预期效果验证
1. **输入延迟**：在大文档（1万行+）中连续输入，主线程阻塞时间应保持在 16ms 以内。
2. **内存占用**：预览区的 DOM 节点数不再随文档长度增长。
3. **滚动性能**：FPS 应稳定在 60 帧，无明显的“留白”或“跳变”现象。

---

## 6. 注意事项
- **组件共享**：所有的 `MarkdownFragment` 应该共享同一套 `remarkPlugins` 和 `components` 配置，避免重复初始化。
- **样式穿透**：由于被拆分成了多个 `div`，原本依赖于 CSS 相邻选择器的样式（如 `h1 + p`）可能需要微调。
- **跨块解析**：如果文档结构（如三级列表）跨越了片段，切分算法需要支持“上下文保留”。建议初步按 `H1/H2` 作为切分锚点。
