## HaoMD 项目论文选题与期刊规划说明

### 一、整体定位：项目的“学术画像”

**系统本体：HaoMD**

HaoMD 是一个基于 Tauri 的桌面 Markdown / 文档工作台，当前和规划的能力大致包括：

- 支持本地文件、多 Tab 管理的 Markdown 工作空间；
- 集成多模型 AI Chat（含 Dify 等），可对当前文档、选区、PDF 等进行问答与生成；
- 已实现 **按 tab 的 AI 会话管理与持久化**，规划中有 **按 docPath 的文档级会话历史**；
- 支持 PDF.js 批注，并与 AI 一起完成摘要、提取、重写等操作；
- 在桌面（Tauri）环境下整合文件系统访问、配置文件、剪贴板、图片保存等能力。

**可包装成论文贡献的核心点**：

- **会话管理与历史：**
  - 按 tab 的会话持久化 + 清理机制（`AIChat会话管理方案.md`）；
  - 按文档路径 docPath 聚合的文档级会话历史（`AIChat文档会话历史方案.md`）；
  - Dify conversationId 与本地 DocConversationRecord 的协同设计。
- **工作流设计：**
  - PDF 批注 + AI Chat + Markdown 笔记的一体化闭环，覆盖“阅读—思考—写作”；
  - 针对知识工作者/研究者的长文档阅读、笔记整理和写作场景。
- **工程与性能：**
  - Tauri 桌面架构下的 AI 状态管理和文件系统持久化；
  - 长会话场景下的内存优化（消息窗口化、历史压缩等）。

这一套可以被视为：**AI 辅助文档/知识工作系统**，既可往 HCI/智能系统方向写，也能往软件工程/工具实践方向写。

### 二、适合的中文核心期刊与选题方向

> 这里列的是对“系统 + 应用 + 一定技术设计”比较友好的期刊，最终是否为中文核心以当年正式目录为准。

#### 1. 《计算机应用》

- **期刊特点：** 综合应用型，系统设计 + 工程实现 + 实验评估的文章比较多，对工具/平台类工作友好。

- **适合的选题方向示例：**

  - **选题 A1：面向知识工作者的 AI 驱动文档工作平台设计与实现**
    - 内容重点：
      - 系统整体架构（Tauri 桌面 + 前端模块划分）；
      - AI Chat 按 tab 会话管理与持久化策略；
      - 文档级会话历史的设计思路与存储结构；
      - 与传统 Markdown 编辑器 / 无 AI 工具在功能或效率上的对比实验。

  - **选题 A2：基于文档级会话历史的智能写作辅助系统设计与实现**
    - 内容重点：
      - 以 `AIChat文档会话历史方案` 为主线；
      - DocConversationRecord / DocConversationMessage / ConversationIndexEntry 的数据模型；
      - conversations_index.json / conversations_data.json 的索引–数据结构设计；
      - 如何利用文档级历史支持写作续写、改写、跨模型上下文共享；
      - 在长文档写作场景下的实验评估。

  - **选题 A3：面向 PDF 文档阅读与知识提取的 AI 辅助批注系统**
    - 内容重点：
      - PDF.js 批注模块设计；
      - “批注 + AI 问答/总结 + Markdown 导出”的工作流；
      - 在信息抽取/文献阅读场景下的效率与质量提升实验；
      - 性能与交互体验分析。

#### 2. 《计算机工程》《计算机工程与应用》等应用/工程向期刊

这类期刊偏“系统 + 工程实践 + 部分优化”，更适合强调架构与性能。

- **选题 B1：AI 辅助 Markdown 编辑器的会话管理与性能优化研究**

  - 内容重点：
    - 按 tab 与按 docPath 的双层会话管理模型；
    - 会话持久化机制（localStorage + Tauri 文件存储）的设计；
    - 消息窗口化、历史压缩等内存优化策略；
    - 大文档与长对话场景下的性能实验（内存占用、响应时间等）；
    - 与“不做优化”或“简单实现”的对比分析。

- **选题 B2：基于 Tauri 的跨平台 AI 文档工作台设计与实现**

  - 内容重点：
    - Tauri 架构下前端与 Rust 后端的职责划分；
    - 文件系统访问、配置文件（ai_settings / prompt_settings）、会话历史的持久化方案；
    - 原生菜单与前端命令系统的集成（File / AI 菜单）；
    - 与纯 Web 实现的资源占用、安全性、可访问本地资料能力对比。

#### 3. 信息/图情类中文核心（如果走知识管理方向）

如《情报学报》《情报科学》等，更关注“信息行为、知识管理、工具对工作流的影响”。

- **选题 C1：AI 驱动的学术阅读与笔记工具对科研工作流的影响研究**

  - 内容重点：
    - 将 HaoMD 定位为科研人员/研究生的阅读与笔记工具；
    - 结合 PDF 批注、AI 摘要、笔记组织等功能；
    - 通过用户实验/访谈分析：
      - 阅读效率、信息提取质量是否提升；
      - 用户对 AI 辅助的接受度与信任变化；
    - 讨论对知识管理与信息行为的启示。

  - 强调智能系统在实际问题中的应用，对系统型工作非常友好。

    - 研究重点：
      - 将 HaoMD 作为一个“智能文档工作台”；

      - 文档级会话历史与 PDF 批注的整合；

      - 任务：阅读 PDF、整理笔记、写摘要/报告；

### 七、关于“AI 助力文档/写作”选题的意义

**1. 宏观层面：生产力工具范式正在迁移**

- Copilot、Notion AI、各种写作助手已经说明：
  - “AI + 文档/代码/笔记”正在改变知识工作方式；
  - 学界与业界都在追问：**如何设计 AI 工具，才能真正提升效率，而不是制造信息噪音和负担？**
- HaoMD 聚焦在“本地文档 + 多 Tab + PDF + AI 会话”的组合场景，是一个典型的知识工作场景载体。

**2. 你的切入点不只是“加个聊天框”，而是抓住了几个关键痛点**

- 会话组织问题：
  - 多 tab、多文档、多模型下，会话如何按 tab / 按文档组织，避免历史散落在无关对话里？
  - 文档级会话历史（按 docPath 聚合）能否改善用户的记忆负担和复盘体验？
- 工作流整合问题：
  - PDF 阅读、批注、AI 问答、Markdown 笔记如何形成闭环，而不是互相割裂？
- 性能与可用性问题：
  - 长会话、长文档场景中，如何通过消息窗口、历史压缩等手段控制内存和界面复杂度？

这些都是当前大量 AI 工具（在线编辑器、笔记应用、浏览器插件）普遍存在的短板，具有明确的现实痛点和改进空间。

**3. 研究意义在于“设计方法 + 实证证据”，而不仅是“功能展示”**

- 真正具有学术价值的是：
  - 提出了一套**文档中心的 AI 会话历史与会话管理设计范式**；
  - 给出清晰的设计原则（例如：按文档聚合、分层持久化、会话压缩与清空策略、与外部服务 ID 的映射方式等）；
  - 通过实验/用户研究证明：
    - 与普通“聊天窗口式” AI 相比，这种设计在特定任务上能显著提升效率、结果质量或主观体验；
    - 或者在某些维度（例如可控性、可复盘性、认知负担）上有明显优势。
- 这类结果对后续设计 AI 写作工具、AI IDE、AI 笔记/知识库系统的研究者和工程师都有参考价值。

**4. 需要避免的风险与“坑”**

- 方向热门但赛道拥挤：
  - 如果仅停留在“我们实现了一个 AI 工具”，容易缺乏新颖性；
  - 需要清晰的研究问题、对比对象和规范的实验设计，才能在 SCI / 核心期刊中脱颖而出。
- 选题范围宜“收窄 + 具体”：
  - 建议从“大而泛的 AI 助力文档工作”中收敛为 1–2 个具体问题，例如：
    - 文档级会话历史对长文档理解/复盘的影响；
    - 按 tab/doc 的会话管理对多任务写作体验与错误率的影响；
    - PDF 批注 + AI 对信息抽取任务的效率提升。
  - 问题一旦具体，论文的“意义”和“贡献”就会更清晰，也更容易设计实验和论证。

**5. 结论：这是一个有前沿性和应用价值的选题方向**

- 只要：
  - 把 HaoMD 当作“研究载体”，而不是单纯“产品展示”；
  - 提炼出 1–2 个聚焦的研究问题；
  - 通过系统设计 + 实验/用户研究给出有说服力的答案；
- 那么：
  - 这个选题对于中文核心期刊和 SCI 应用类期刊，都是有实际意义和发表潜力的。
